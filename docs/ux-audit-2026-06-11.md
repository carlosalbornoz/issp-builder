## 🔴 1. Dialogs don't scroll — content clips off-screen on short viewports

**Root cause:** `src/components/ui/dialog.tsx:53-60` — the base `DialogContent` is
`fixed top-1/2 left-1/2 -translate-y-1/2` with **no `max-height` and no overflow**. When the
content is taller than the viewport, it's vertically centered and the top + bottom clip off the
screen with no way to reach them. This is exactly the reported tablet bug (tablet landscape is
short — ~768–820px — and the properties form is long).

**Affected live dialogs:**
- `src/components/editor/issp-properties-dialog.tsx:383` — `sm:max-w-lg` (the reported one; the
  agency form + logo control + coverage/scope sections easily exceed a landscape tablet height)
- `src/components/editor/new-issp-dialog.tsx:71` — same long shared form
- `src/components/documents/create-document-dialog.tsx:105` — dormant (server-side tree) but same flaw

The What's-New modals (`home-page-client.tsx:89,455`) *don't* have the bug because they manually
add `max-h-[82vh] flex flex-col p-0 gap-0` + an inner `overflow-y-auto`. That pattern should live
in the base component.

**Fix (recommended):** give `DialogContent` a default `max-h-[calc(100dvh-2rem)]` and make it a
flex column, with the body scrollable. Cleanest is a small structural change: header/footer stay
fixed, the middle scrolls. Minimal version — add to the base popup className:
`max-h-[calc(100dvh-2rem)] overflow-y-auto`. Better version — `flex max-h-[calc(100dvh-2rem)]
flex-col` on the popup and let a body wrapper scroll (mirrors the What's-New modals and keeps the
footer buttons always reachable). Use `100dvh` (not `vh`) for mobile browser chrome.

## 🔴 2. Number inputs trigger the full keyboard on touch devices

**Files:** 17 `type="number"` inputs across the editor; **zero** set `inputMode`. Heaviest in
Part IV budget (`part4-year-form.tsx:216,225,465,474`) and Part I-B headcount / Part III-C.

On tablets and phones, `type="number"` without `inputMode` often still surfaces a full QWERTY
keyboard (or an inconsistent one). Budget and headcount entry is the most numeric-heavy part of
the whole tool, so this is real friction.

**Fix:** add `inputMode="numeric"` (integers: qty, headcount, adoption %) or `inputMode="decimal"`
(currency: unitCost) to each numeric field. Consider a shared `<NumberInput>` wrapper so this is
applied consistently and future fields inherit it.

## 🟡 3. Touch targets are below the 44px minimum

**Files:** `src/components/ui/button.tsx:22-33`, `src/components/ui/input.tsx:12`

Defaults: inputs `h-8` (32px), default button `h-8` (32px), `icon` `size-8` (32px), `icon-sm`
`size-7` (28px), `sm` `h-7` (28px), `xs` `h-6` (24px). Apple HIG recommends ≥44px and Material
≥48px for touch. The compact sizing is deliberate for desktop density (a stated preference), but on
a tablet the destructive icon buttons in the stakeholder/budget drawers (`h-8 w-8`, `size-7`) and
the row-delete buttons are easy to mis-tap.

**Fix:** don't inflate everything — add a coarse-pointer affordance. Either bump interactive
controls under `@media (pointer: coarse)` (e.g. min-height 40–44px), or specifically enlarge the
icon/delete buttons that live inside drawers and table rows. Keep desktop density as-is.

## 🟡 4. Number fields can't be cleared and show a sticky "0"

**File:** `src/components/issp-editor/part4/part4-year-form.tsx:219,229,469,479`

- `set("qty", Math.max(1, Number(e.target.value)))` — clearing the field makes `Number("")` → `0`,
  then `Math.max(1, 0)` → `1`, so the input snaps to `1` mid-edit and you can't clear-to-retype.
- `set("unitCost", Number(e.target.value))` — empty → `0`; the controlled `value={0}` renders a
  literal `0` the user must delete before typing, and typing after it produces values like `0500`.

This pattern repeats anywhere a numeric value is stored as a `number` and echoed back. It's a small
but constant annoyance in the budget tables.

**Fix:** keep the raw string in form state while editing and coerce on blur/save, or allow an empty
string and only clamp on commit. A shared `<NumberInput>` (see #2) is the natural home for this.

## 🟡 5. Properties dialog resets the form if the doc changes while open

**File:** `src/components/editor/issp-properties-dialog.tsx:262-276`

The `useEffect` that seeds the form from `doc` depends on `[open, doc]`. Because autosave rewrites
`doc` (new `updatedAt`) on every keystroke elsewhere, any background store update while the dialog
is open re-runs the effect and overwrites in-progress edits. In practice the dialog is modal so
concurrent edits are unlikely, but it's fragile (and is the lingering `set-state-in-effect`
eslint-disable). Same render-phase-reset pattern used in the Part I-C drawer fix applies here.

## ⚪ 6. Latent: bottom/top Sheet variants have no max-height

**File:** `src/components/ui/sheet.tsx:56` — `data-[side=bottom]`/`data-[side=top]` use `h-auto`
with no `max-height` or overflow. All current usages are `side="right"` (which is fine), so this is
latent, but a future bottom-sheet with long content would overflow. Add
`max-h-[calc(100dvh-2rem)] overflow-y-auto` to those variants when first used.

## ⚪ 7. Logo upload has no read/processing feedback

**File:** `src/components/editor/issp-properties-dialog.tsx` (logo handler)

`readFileAsDataUrl` is async with no pending state. For a 2 MB image the preview just pops in after
a beat with no spinner. Low impact (files are small), but a brief disabled/loading state on the
button would feel more responsive.
